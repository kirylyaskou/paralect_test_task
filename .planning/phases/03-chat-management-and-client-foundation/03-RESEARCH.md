## Summary

Phase 3 builds the chat management layer and application shell on top of the existing DB access layer (Phase 1) and authentication system (Phase 2). The core work is: (1) three API route handlers (GET/POST chats, DELETE chat by id) that use the existing `lib/db/chats.ts` functions with session-based ownership validation, (2) TanStack Query hooks with optimistic update patterns for create/delete mutations, (3) a shadcn Sidebar-based responsive layout replacing the current empty `(main)/layout.tsx`, and (4) a theme toggle using the already-configured `next-themes` provider.

All building blocks are already in place. The DB layer (`lib/db/chats.ts`) has full CRUD, the auth helper (`lib/auth/helpers.ts`) provides `requireAuth()`, Zod schemas (`lib/schemas/chat.ts`) exist, types (`lib/types/index.ts`) export `Chat`, and the `ThemeProvider` is already wired in `providers.tsx`. The shadcn sidebar component (v4 base-nova style) needs to be installed along with dialog, dropdown-menu, and scroll-area. No new npm packages beyond shadcn components are needed.

**Primary recommendation:** Follow the established 3-layer pattern strictly -- API routes call `requireAuth()` + DB functions, client hooks call API routes via `fetch()`, components consume hooks. Use TanStack Query v5 `useMutation` with `onMutate`/`onError`/`onSettled` for optimistic updates. Use shadcn's built-in `SidebarProvider` + `Sidebar` + `SidebarInset` for responsive layout.

### Pattern 1: API Route Handler with Auth + Ownership

**What:** Standard pattern for all chat API routes -- authenticate via session, validate ownership, call DB layer, return JSON.

**When to use:** Every API route in this phase.

**Example (verified against Next.js 16.2.1 docs in `node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/route.md`):**

```typescript
// app/api/chats/[id]/route.ts
import { NextRequest } from 'next/server'
import { requireAuth } from '@/lib/auth/helpers'
import { getChatById, deleteChat } from '@/lib/db/chats'
import { AuthenticationError, DatabaseError } from '@/lib/errors'

export async function DELETE(
  _request: NextRequest,
  { params }: { params: Promise<{ id: string }> }
) {
  try {
    const { userId } = await requireAuth()
    const { id } = await params  // Next.js 16: params is a Promise

    // Ownership check (CHAT-05)
    const chat = await getChatById(id)
    if (chat.user_id !== userId) {
      return Response.json({ error: 'Forbidden' }, { status: 403 })
    }

    await deleteChat(id)
    return Response.json({ success: true })
  } catch (error) {
    if (error instanceof AuthenticationError) {
      return Response.json({ error: 'Unauthorized' }, { status: 401 })
    }
    if (error instanceof DatabaseError) {
      return Response.json({ error: 'Chat not found' }, { status: 404 })
    }
    return Response.json(
      { error: 'Something went wrong. Please try again.' },
      { status: 500 }
    )
  }
}
```

**CRITICAL Next.js 16 detail:** The `params` in route handlers is a `Promise` that must be awaited. This is confirmed in `node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/route.md` -- `{ params }: { params: Promise<{ id: string }> }`.

### Pattern 2: TanStack Query v5 Optimistic Mutation

**What:** Mutation hook that immediately updates the query cache before server response, with rollback on error.

**When to use:** Chat create and delete operations.

**Example:**

```typescript
// hooks/use-chats.ts
import { useMutation, useQuery, useQueryClient } from '@tanstack/react-query'
import type { Chat } from '@/lib/types'

export function useChats() {
  return useQuery<Chat[]>({
    queryKey: ['chats'],
    queryFn: async () => {
      const res = await fetch('/api/chats')
      if (!res.ok) throw new Error('Failed to fetch chats')
      const data = await res.json()
      return data.chats
    },
  })
}

export function useDeleteChat() {
  const queryClient = useQueryClient()

  return useMutation({
    mutationFn: async (chatId: string) => {
      const res = await fetch(`/api/chats/${chatId}`, { method: 'DELETE' })
      if (!res.ok) throw new Error('Failed to delete chat')
    },
    onMutate: async (chatId) => {
      // Cancel outgoing refetches
      await queryClient.cancelQueries({ queryKey: ['chats'] })

      // Snapshot previous value
      const previousChats = queryClient.getQueryData<Chat[]>(['chats'])

      // Optimistically remove from cache
      queryClient.setQueryData<Chat[]>(['chats'], (old) =>
        old?.filter((chat) => chat.id !== chatId)
      )

      return { previousChats }
    },
    onError: (_err, _chatId, context) => {
      // Rollback on error
      if (context?.previousChats) {
        queryClient.setQueryData(['chats'], context.previousChats)
      }
      console.error('Failed to delete chat. Please try again.')
    },
    onSettled: () => {
      // Refetch to ensure consistency
      queryClient.invalidateQueries({ queryKey: ['chats'] })
    },
  })
}
```

## Common Pitfalls

### Pitfall 1: Forgetting to Await params in Next.js 16

**What goes wrong:** Using `params.id` directly without `await` causes "Cannot read properties of Promise" error.
**Why it happens:** Next.js 15+ changed `params` from a plain object to a `Promise<object>` in both pages and route handlers.
**How to avoid:** Always destructure after awaiting: `const { id } = await params`
**Warning signs:** TypeScript errors about Promise missing expected properties.

### Pitfall 2: Optimistic Create Without Temp ID

**What goes wrong:** Creating a chat optimistically requires a temporary ID for the sidebar item, but the real ID comes from the server.
**Why it happens:** POST /api/chats returns the real chat with a UUID. The optimistic entry needs a placeholder.
**How to avoid:** Use `crypto.randomUUID()` as temp ID in `onMutate`, then `invalidateQueries` in `onSettled` replaces it with real data. Navigate to `/chat/[realId]` only after the mutation succeeds (in `onSuccess`, not `onMutate`).
**Warning signs:** Navigation to a temp ID URL that does not exist on the server.

### Pitfall 3: Layout Component Must Be Server Component

**What goes wrong:** Adding `'use client'` to `(main)/layout.tsx` to use hooks directly.
**Why it happens:** Need to pass client-side state (like sidebar open state) -- but layouts should remain Server Components.
**How to avoid:** `SidebarProvider` is already a client component. The layout itself stays as a Server Component, just renders `<SidebarProvider>` which handles all client state internally. Components inside (AppSidebar, ContentHeader) are client components.
**Warning signs:** Unnecessary `'use client'` on layout files.

### Pitfall 4: Ownership Check Race Condition on Delete

**What goes wrong:** Deleting a chat without first verifying ownership allows horizontal privilege escalation.
**Why it happens:** The DB layer `deleteChat(chatId)` does not check ownership -- it deletes any chat by ID.
**How to avoid:** API route must `getChatById(id)` first, compare `chat.user_id !== userId`, return 403. The DELETE route handler owns this responsibility.
**Warning signs:** Missing ownership check in delete route.

### Pitfall 5: SidebarProvider Must Be Within ThemeProvider

**What goes wrong:** Theme-aware sidebar colors don't update when toggling theme.
**Why it happens:** SidebarProvider relies on CSS variables that change with the `.dark` class. If it's outside the ThemeProvider scope, it won't react to changes.
**How to avoid:** The `Providers` component in `app/providers.tsx` already wraps children in `ThemeProvider`. Since `(main)/layout.tsx` is rendered inside `Providers` via the root layout, `SidebarProvider` will automatically be within `ThemeProvider` scope. No special action needed -- just verify the component hierarchy.
**Warning signs:** Theme toggle changes content area but not sidebar.

### Pitfall 6: Sheet Accessibility in Shadcn Sidebar

**What goes wrong:** Accessibility warnings about missing `DialogTitle` or `SheetTitle` when sidebar opens as Sheet on mobile.
**Why it happens:** The shadcn sidebar component internally uses Sheet for mobile, which requires a title for screen readers.
**How to avoid:** The shadcn sidebar already handles this internally with `sr-only` SheetTitle. No extra work needed. But if customizing, ensure the Sheet always has a title.
**Warning signs:** Console warnings about missing dialog/sheet title.

### API Route: GET /api/chats (List User's Chats)

```typescript
// app/api/chats/route.ts
import { NextRequest } from 'next/server'
import { requireAuth } from '@/lib/auth/helpers'
import { getChatsByUserId, createChat } from '@/lib/db/chats'
import { createChatSchema } from '@/lib/schemas/chat'
import { AuthenticationError, DatabaseError } from '@/lib/errors'

export async function GET() {
  try {
    const { userId } = await requireAuth()
    const chats = await getChatsByUserId(userId)
    return Response.json({ chats })
  } catch (error) {
    if (error instanceof AuthenticationError) {
      return Response.json({ error: 'Unauthorized' }, { status: 401 })
    }
    if (error instanceof DatabaseError) {
      return Response.json({ error: 'Failed to fetch chats' }, { status: 500 })
    }
    return Response.json(
      { error: 'Something went wrong. Please try again.' },
      { status: 500 }
    )
  }
}

export async function POST(request: NextRequest) {
  try {
    const { userId } = await requireAuth()
    const body = await request.json()

    const result = createChatSchema.safeParse(body)
    if (!result.success) {
      return Response.json(
        { error: 'Validation failed', details: result.error.flatten() },
        { status: 400 }
      )
    }

    const chat = await createChat(userId, result.data.title)
    return Response.json({ chat }, { status: 201 })
  } catch (error) {
    if (error instanceof AuthenticationError) {
      return Response.json({ error: 'Unauthorized' }, { status: 401 })
    }
    if (error instanceof DatabaseError) {
      return Response.json({ error: 'Failed to create chat' }, { status: 500 })
    }
    return Response.json(
      { error: 'Something went wrong. Please try again.' },
      { status: 500 }
    )
  }
}
```

### Optimistic Create Chat Mutation

```typescript
export function useCreateChat() {
  const queryClient = useQueryClient()
  const router = useRouter()

  return useMutation({
    mutationFn: async () => {
      const res = await fetch('/api/chats', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({}),
      })
      if (!res.ok) throw new Error('Failed to create chat')
      const data = await res.json()
      return data.chat as Chat
    },
    onMutate: async () => {
      await queryClient.cancelQueries({ queryKey: ['chats'] })
      const previousChats = queryClient.getQueryData<Chat[]>(['chats'])

      // Optimistic entry with temp data
      const tempChat: Chat = {
        id: crypto.randomUUID(),
        user_id: '',
        title: null,
        created_at: new Date().toISOString(),
        updated_at: new Date().toISOString(),
      }

      queryClient.setQueryData<Chat[]>(['chats'], (old) =>
        [tempChat, ...(old ?? [])]
      )

      return { previousChats }
    },
    onSuccess: (newChat) => {
      // Navigate to real chat after server confirms
      router.push(`/chat/${newChat.id}`)
    },
    onError: (_err, _vars, context) => {
      if (context?.previousChats) {
        queryClient.setQueryData(['chats'], context.previousChats)
      }
      console.error('Failed to create chat. Please try again.')
    },
    onSettled: () => {
      queryClient.invalidateQueries({ queryKey: ['chats'] })
    },
  })
}
```
